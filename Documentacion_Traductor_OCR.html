1. Introducción

El presente documento describe el diseño, funcionamiento y uso del proyecto Traductor-OCR, una aplicación desarrollada como parte del curso de Traductores y Compiladores. El sistema toma como entrada imágenes o documentos PDF de facturas reales y produce, como salida, una representación estructurada (JSON) con los campos relevantes del comprobante: número de factura, proveedor, fecha, RFC y total.

Aunque el reconocimiento óptico de caracteres (OCR) suele asociarse con tareas de visión por computadora, el problema que aborda este proyecto es esencialmente traductológico: tomar un texto fuente —en este caso, un texto deformado por errores de OCR— y producir un texto destino que cumpla con una gramática bien definida (la estructura JSON). En ese sentido, el proyecto aplica las etapas clásicas de un traductor: análisis léxico (regex sobre el texto reconocido), análisis sintáctico (validación de patrones), y generación del código objetivo (serialización a JSON).

La aplicación expone una interfaz web construida con Streamlit que permite cargar facturas, revisarlas manualmente, almacenarlas y realizar agregaciones de totales sobre el conjunto de documentos guardados.

2. Descripción general del sistema

El Traductor-OCR está implementado en Python 3.13 y se apoya en las siguientes tecnologías principales:

ComponenteTecnologíaFunción
InterfazStreamlitUI web para carga, revisión, edición y agregación.
PreprocesamientoOpenCV, NumPyMejora de contraste, eliminación de ruido y corrección de inclinación.
OCREasyOCR (es + en)Reconocimiento de caracteres sobre imágenes.
Extracción PDFpdfplumberExtracción directa de texto y tablas de PDFs nativos.
AnálisisExpresiones regulares (re)Identificación de campos estructurados (folio, RFC, total, etc.).
PersistenciaJSON localAlmacenamiento por archivo en data/process/.

El sistema acepta archivos en los formatos PNG, JPG, JPEG, BMP, TIFF y PDF. Los formatos de imagen se enrutan al pipeline de preprocesamiento + OCR, mientras que los PDFs se enrutan a un extractor que aprovecha el texto embebido del documento (cuando éste es nativo y no escaneado).

    ├── reader.py              # OCR con EasyOCR
    ├── extractor_pdf.py       # Extracción con pdfplumber

3.2. Flujo de procesamiento

El recorrido de un archivo desde su carga hasta su almacenamiento estructurado sigue las siguientes etapas, según el tipo de entrada:

Paso 1 — Recepción. El usuario arrastra uno o varios archivos a la pestaña «Subir y revisar». Streamlit los sube a memoria temporal.
Paso 2a — Ruta de imagen. Si el archivo es una imagen, se guarda como temporal en disco, se ejecuta preprocess.process_image() (CLAHE + denoising + deskew) y luego reader.extract_text_image() aplica EasyOCR para obtener texto crudo.
Paso 2b — Ruta de PDF. Si el archivo es PDF, se llama directamente a extractor_pdf.extract_invoice_pdf(), que usa pdfplumber para extraer texto y tablas de cada página.
Paso 3 — Corrección. El texto crudo pasa por corrector.correct_text(), que normaliza espacios y sustituye caracteres comúnmente confundidos por OCR (O→0, l→1, S→5, etc.) dentro de patrones numéricos monetarios.
Paso 4 — Análisis léxico. parser.parse_invoice() aplica un conjunto de expresiones regulares para localizar los campos: número de factura, proveedor, fecha, RFC y total. Para el total, se usa una heurística línea-por-línea que descarta subtotales, IVA, descuentos y propinas.
Paso 5 — Revisión humana. La interfaz muestra los datos extraídos en una tarjeta. El usuario puede corregir el total manualmente si no se detectó, ver el OCR completo, y decidir guardar o descartar.
Paso 6 — Persistencia. storage.save_invoices() serializa el diccionario a JSON dentro de data/process/<nombre>.json.
Paso 7 — Agregación. Las pestañas «Facturas guardadas» y «Calculadora» leen los JSON con storage.load_invoices() y aplican aggregator.sum_all() o sum_selection() para computar totales globales o parciales.

4. Manual de uso

4.1. Requisitos previos

  • Python 3.10 o superior (probado con 3.13).
  • Sistema operativo: Windows, macOS o Linux.
  • Conexión a Internet la primera vez que se ejecuta EasyOCR (para descargar los modelos de los idiomas es y en).
  • Aproximadamente 500 MB de espacio en disco para los modelos OCR.

4.2. Instalación

Desde una terminal, dentro de la carpeta del proyecto, se recomienda crear un entorno virtual y luego instalar las dependencias:

# 1. Crear entorno virtual
python -m venv venv

# 2. Activar
# En Windows:
venv\Scripts\activate
# En macOS/Linux:
source venv/bin/activate

# 3. Instalar dependencias
pip install streamlit easyocr opencv-python numpy pdfplumber
Nota: No existe un requirements.txt en el proyecto. Se recomienda generarlo después de la primera instalación con pip freeze > requirements.txt para facilitar la reproducibilidad de la entrega.

4.3. Ejecución

La aplicación se lanza con Streamlit apuntando al archivo principal:

streamlit run interface/app.py

Streamlit abrirá automáticamente el navegador en http://localhost:8501. La aplicación queda en ejecución hasta que se cierre la terminal o se presione Ctrl+C.

4.4. Operación de la interfaz

La aplicación se compone de tres pestañas, cada una con un propósito específico:

Pestaña 1 — «Subir y revisar»

Es el punto de entrada del flujo. Permite arrastrar o seleccionar uno o varios archivos en formatos PNG, JPG, JPEG, BMP, TIFF o PDF. Para cada archivo procesado se genera una tarjeta de revisión con:

  • El nombre del archivo y el total detectado en grande (en color dorado).
  • Una insignia que indica si el total fue detectado automáticamente o si requiere intervención manual.
  • Los campos extraídos: proveedor, fecha, número de factura, RFC y folio.
  • Un campo de entrada manual para el total cuando el parser no logró detectarlo.
  • Un panel desplegable («Ver texto OCR completo») que muestra el resultado bruto del OCR junto con la línea exacta usada para identificar el total (útil para depurar la heurística).
  • Botones Guardar y Descartar, así como un botón Guardar todas las pendientes al final de la lista.

Pestaña 2 — «Facturas guardadas»

Lista todas las facturas almacenadas como archivos JSON. Muestra tres métricas en la parte superior: número de documentos, suma total y proporción de facturas con total detectado. Por cada fila se ofrecen acciones de eliminación y edición. Al editar, los campos modificables son proveedor, número de factura, fecha y total. Existe además una zona de peligro para eliminar el conjunto completo.

Pestaña 3 — «Calculadora»

Permite seleccionar manualmente un subconjunto de facturas guardadas y ver simultáneamente el subtotal de la selección y el total global. Bajo la selección se despliega el detalle de cada factura incluida.

5. Documentación técnica por módulo

5.1. interface/app.py — Aplicación principal

Es el orquestador completo de la aplicación. Concentra la configuración de página, los estilos CSS personalizados (paleta oscura con acentos dorados), el manejo de estado de sesión y las tres pestañas funcionales.

Responsabilidades principales:

  • Inserta dependencias del proyecto al sys.path para permitir la importación de los módulos hermanos.
  • Configura la página (st.set_page_config) con título Facturas OCR, ícono de recibo y layout amplio.
  • Inyecta una hoja de estilos extensa que personaliza tipografías (IBM Plex Sans/Mono), colores (fondo #0F0F0F, acento #C8B560), tarjetas, tabs, botones, expanders y métricas.
  • Inicializa el estado de sesión con dos estructuras: st.session_state.pendientes (lista de facturas en revisión) y st.session_state.procesados (set de nombres ya procesados, para evitar reprocesamiento).
  • Despacha la lógica de las tres pestañas declaradas con st.tabs([...]).

Lógica de la pestaña «Subir y revisar»:

El uploader acepta múltiples archivos. Para cada archivo nuevo, se decide la ruta:

if suffix == ".pdf":
    text = extract_invoice_pdf(file)
else:
    tmp = tempfile.NamedTemporaryFile(...)
    tmp.write(file.read())
    img = process_image(tmp.name)
    text = extract_text_image(img)

corrected = correct_text(text)
data = parse_invoice(corrected)

El resultado se almacena en st.session_state.pendientes, lo que permite revisar varios documentos a la vez sin perder el estado al re-renderizar.

Lógica de la pestaña «Facturas guardadas»:

Implementa un CRUD completo: lectura con load_invoices(), eliminación individual con delete_invoice(idx), edición con get_invoice(idx) + update_invoice(idx, ...) y eliminación masiva en una zona de peligro plegable.

Lógica de la pestaña «Calculadora»:

Reusa aggregator.sum_all() y aggregator.sum_selection() para calcular totales globales y parciales. La selección usa st.multiselect con un format function que muestra «número · proveedor — $monto».

5.2. modules/preprocess.py — Preprocesamiento de imagen

Encapsula todo el preprocesado previo al OCR. La función pública es process_image(path) -> np.ndarray. El módulo distingue entre imágenes a color y en escala de grises, y aplica pipelines distintos.

FunciónPropósito
_is_colorful(img)Decide si la imagen tiene color significativo evaluando la saturación media en HSV (umbral 25).
_enhance_bw(img)Para imágenes B/N: convierte a gris, aplica fastNlMeansDenoising y luego CLAHE (ecualización adaptativa) con clipLimit=2.0.
_enhance_color(img)Para imágenes a color: reescala a un mínimo de 1000 px en su lado mayor, aplica bilateralFilter (preserva bordes), convierte a gris y CLAHE.
_deskew(img)Corrige inclinaciones menores a ~10°. Usa cv2.minAreaRect sobre los píxeles oscuros y rota la imagen con warpAffine. Salta correcciones <0.5° por economía.
process_image(path)Punto de entrada. Lee el archivo, lo redimensiona si excede 3000 px, elige el pipeline según el color, y aplica deskew al final.

El módulo está pensado para ser robusto frente a fotografías reales de facturas (que suelen estar inclinadas, mal iluminadas o tener contraste irregular).

5.3. modules/reader.py — Motor OCR

Wrapper sobre EasyOCR que entrega texto reconocido reorganizado por líneas según la posición vertical de cada caja delimitadora.

Inicialización:

reader = easyocr.Reader(['es', 'en'], gpu=False)

El lector se instancia a nivel de módulo (singleton implícito), por lo que los modelos se cargan en RAM una sola vez y persisten durante toda la sesión.

Función extract_text_image(image):

  1. Llama a reader.readtext(image, detail=1, paragraph=False), que devuelve tuplas (bbox, texto, confianza).
  2. Filtra detecciones con confianza menor a 0.25 (descarta ruido).
  3. Ordena las cajas por su coordenada y superior y, en caso de empate, por x.
  4. Agrupa cajas en líneas usando un umbral adaptativo basado en la altura de cada caja (max(height·0.6, 12 px)): si la siguiente caja está dentro del umbral vertical, pertenece a la misma línea; si no, comienza una nueva.
  5. Dentro de cada línea, ordena las cajas por x y concatena los textos con un espacio.
  6. Devuelve todas las líneas unidas con saltos de línea.

Esta agrupación geométrica es esencial porque EasyOCR devuelve detecciones individuales sin orden de lectura natural; sin esta etapa, los campos de la factura aparecerían mezclados.

5.4. modules/extractor_pdf.py — Extracción de PDF

Cuando el archivo de entrada es un PDF nativo (no una imagen escaneada), evita por completo el OCR y usa pdfplumber para extraer el texto embebido. Esto es órdenes de magnitud más rápido y exacto que aplicar OCR sobre el PDF rasterizado.

Función extract_invoice_pdf(pdf_file_path):

  1. Abre el PDF con pdfplumber.open().
  2. Itera sobre cada página y concatena: el texto extraído con page.extract_text() y, adicionalmente, las celdas no nulas de la tabla detectada con page.extract_table(), unidas en una línea por fila.
  3. Devuelve un único string con el resultado.
Nota técnica: En el código actual se observa una rama else tras el for row in table que vuelve a extraer texto de la página y lo concatena. Esto produce texto duplicado cuando la página tiene tabla. En una próxima iteración convendría eliminar esa rama o controlarla con una bandera.

5.5. modules/corrector.py — Corrector textual

Limpia el texto OCR y corrige confusiones típicas en contextos numéricos.

Mapa de sustituciones (_OCR_NUM_MAP):

OCR confundeRealOCR confundeReal
O, o, D0S5
I, l, i1B8
Z2G6
T7

Procesamiento de correct_text(text):

  1. Divide el texto en líneas.
  2. Por cada línea, elimina caracteres de control (\x00–\x08, \x0B–\x1F, \x7F).
  3. Normaliza secuencias de espacios múltiples a un único espacio.
  4. Detecta patrones que parecen montos monetarios y aplica el mapa de sustitución solo dentro de esos patrones, evitando alterar texto que no sea numérico.
  5. Reune y devuelve el texto.

La elección de aplicar las sustituciones solo en contextos numéricos es importante: una O en «Oxxo» no debe convertirse en cero; pero la misma letra dentro de «$1OO.OO» sí.

5.6. modules/parser.py — Analizador léxico de facturas

Transforma el texto corregido en un diccionario con los campos relevantes. Es, conceptualmente, el analizador léxico del traductor.

Función _parse_money(value)

Convierte cadenas de dinero heterogéneas a float, manejando formato mexicano (1.234,56), formato inglés (1,234.56) y casos sin separador de miles. Su lógica analiza el conteo y el orden relativo de comas y puntos para decidir cuál es el separador decimal.

Función parse_invoice(text)

Aplica un conjunto de expresiones regulares para extraer los siguientes campos:

CampoPatrónEjemplo capturado
numero_facturaFactura/Folio/No./Comprobante seguido de un identificador alfanumérico.«Factura: A-12345»
proveedorProveedor/Emisor/Empresa/De/Sucursal/Razón Social seguido del nombre.«Proveedor: OXXO S.A.»
fechaFecha en formato dd/mm/aaaa o ISO con hora.«30/04/2026», «2026-04-30T15:20:28»
rfc3–4 letras + 6 dígitos + 3 alfanuméricos (RFC mexicano).«RFC: ANA050518RL1»
totalHeurística línea-por-línea descartando subtotales, IVA, descuentos y propinas.1531.34

Heurística para el total: es la parte más sutil del parser. Se recorren las líneas del texto y, para cada una:

  1. Se descartan líneas que contengan «subtotal», «descuento», «iva», «propina» o «cargo» a menos que también contengan la palabra «Total» con bordes de palabra (caso «IVA Total»).
  2. Si la línea contiene «Total» (no parte de otra palabra), se buscan todos los montos con un patrón de dinero robusto que admite separadores de miles y decimales en cualquiera de los dos formatos.
  3. Se toma el último monto de la línea, asumiendo que el total numérico va a la derecha de la etiqueta.
  4. Se conserva además la línea exacta usada en _debug_total_line para que la interfaz pueda exhibirla y facilitar la depuración manual.

5.7. modules/aggregator.py — Agregador de totales

Módulo muy compacto que ofrece dos funciones públicas para sumar totales de manera segura.

FunciónDescripción
_safe_total(invoice)Devuelve el total como float si existe y es numérico; en caso contrario, 0.0.
sum_all(invoices)Suma el total de todas las facturas pasadas.
sum_selection(invoices, index)Suma solo las facturas cuyos índices estén en index.

La separación de esta lógica facilita pruebas unitarias y aísla la suma del resto del flujo.

5.8. modules/storage.py — Persistencia JSON

Implementa un CRUD por archivo: cada factura se guarda como un JSON independiente en data/process/. La ruta se calcula a partir de la ubicación de storage.py, lo que la hace independiente del directorio de ejecución.

FunciónDescripción
_sorted_files()Lista los archivos *.json del directorio en orden alfabético.
save_invoices(data, name_archivo)Crea el directorio si hace falta y serializa el diccionario con indent=4 y UTF-8 sin escapado.
load_invoices()Carga todos los JSON en una lista de diccionarios.
get_invoice(idx)Carga el JSON en la posición idx; devuelve None si está fuera de rango.
update_invoice(idx, new_data)Lee, fusiona campos no vacíos y reescribe el JSON. Convierte el total a float si llega como cadena.
delete_invoice(idx)Elimina el archivo JSON correspondiente.
Observación: en el cálculo de _STORAGE_DIR hay una pequeña duplicación (data/process aparece dos veces en la ruta). En la práctica esto puede generar la creación de una subcarpeta data/process/data/process; en la próxima iteración convendría revisar y simplificar la composición de la ruta.

6. Caso de ejemplo

El proyecto incluye una imagen de prueba (assets/factura_gema.png) y un resultado procesado (data/process/EmilyLeonVuelo.json). El JSON de salida ilustra la estructura final de un documento procesado:

{
    "numero_factura": "152528792",
    "proveedor": "LEYPERSONASMORALES",
    "fecha": "2026-01-30T14:14:20",
    "rfc": "ANA050518RL1",
    "total": 1531.34,
    "_debug_total_line": "Total $1,531.34"
}

Cada campo proviene de una etapa identificable del traductor:

  • numero_factura: capturado por la primera regex del parser.
  • proveedor: extraído del campo «Razón Social». La cadena LEYPERSONASMORALES sugiere que el OCR fusionó tokens; un postprocesado adicional podría restaurar espacios.
  • fecha: en formato ISO, característico de los CFDI mexicanos.
  • rfc: validado contra el patrón [A-Z&Ñ]{3,4}\d{6}[A-Z0-9]{3}.
  • total: detectado por la heurística de línea, conservando la línea origen en _debug_total_line.

7. Conclusiones

El proyecto Traductor-OCR demuestra que un problema de extracción documental puede modelarse con la misma metodología de un traductor clásico: un análisis léxico (regex), una etapa de saneamiento (corrector OCR) y una representación estructurada de salida (JSON). La separación clara en módulos —preprocesamiento, OCR, corrección, parsing, agregación y persistencia— permite probar y mejorar cada etapa de forma independiente.

Como áreas de mejora, se identifican: (1) corregir la duplicación de ruta en storage.py; (2) limpiar la rama else redundante en extractor_pdf.py; (3) generar un requirements.txt formal; (4) añadir pruebas unitarias para parser._parse_money y para la heurística del total; y (5) considerar el reemplazo de las regex por una gramática más estricta para campos especialmente sensibles como RFC y total, lo cual acercaría aún más el sistema al modelo formal de un traductor con un analizador sintáctico explícito.
